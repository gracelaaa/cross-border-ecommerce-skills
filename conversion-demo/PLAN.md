# 竞品对标转化诊断 Demo 规划

> 版本：v0.1｜目标：先做出可展示、可收集真实商家、可验证付费意愿的 Demo，而不是直接开发完整 SaaS。

## 1. 结论：Demo 要证明什么

Demo 不是为了证明“我们能抓网页”，而是要验证三个核心假设：

1. **成熟一点的独立站商家愿意输入自己的链接和对标链接；**
2. **商家认为“自有 vs 对标”的差异比泛泛的单站建议更有价值；**
3. **商家愿意根据具体建议改页面，甚至为完整诊断付费。**

因此第一版采用：

```text
前端可操作 Demo + 公开页面分析 + 人工复核 + 报告预览
```

底层可以是人工和脚本，用户不需要知道后面是不是全自动。我们先验证价值，再自动化重复环节。

---

## 2. Demo 的目标用户

首批不面向“想找爆品”的小白，而面向：

- 已经有 Shopify 或独立站；
- 近 30 天至少 1,000 次访问，优先 3,000+；
- 已经投放 Meta、Google、TikTok，或有稳定自然流量；
- 有 1–3 个主推商品；
- 能提供店铺链接和至少部分经营数据；
- 正在考虑优化商品页、首页或广告落地页。

Demo 表单需要用访问量和订单数做筛选，避免大量“没有流量、只想免费咨询”的线索进入。

---

## 3. 用户端完整流程

```text
访问落地页
  ↓
看 60 秒示例
  ↓
点击“申请一次对比诊断”
  ↓
填写自有链接 + 对标链接 + 基础数据
  ↓
系统校验链接并显示分析范围
  ↓
进入分析中状态
  ↓
查看 Demo 版结果/等待人工复核
  ↓
看到 Top 3 问题和一个对比矩阵
  ↓
留下邮箱/联系方式领取完整报告
  ↓
预约 15–20 分钟反馈
```

### 推荐两种模式

#### 模式 A：半自动真实分析（首选）

用户提交真实链接后，你人工触发分析，完成后发送报告链接。

优点：

- 结论质量可控；
- 适合早期真实用户；
- 不需要先解决所有反爬、动态渲染和异常页面问题。

#### 模式 B：即时示例 Demo

用户输入任意链接，页面只展示预设的示例结果或有限的自动抽取结果。

优点：

- 能让访客即时体验；
- 适合落地页转化和广告素材。

限制：

- 不能把模拟结果伪装成真实诊断；
- 页面必须明确标记“演示结果”或“待人工复核”。

建议：**两种模式一起做，但真实用户以模式 A 为主。**

## 4. P0：Demo 必须具备的能力

### P0-1. 价值说明落地页

首屏需要讲清楚：

- 给谁用：有流量但转化不理想的独立站商家；
- 输入什么：自己的商品页 + 对标页面；
- 输出什么：差异、问题、改法、收入情景和复测指标；
- 不需要什么：不需要 Shopify 登录、不需要广告账户权限；
- 当前状态：早期测试，结论由公开页面证据和人工复核组成。

建议文案：

> 不只是告诉你页面哪里不好，而是把你的商品页和对标页面放在一起，指出用户关心但你没讲清楚的地方，并给出下一步怎么改。

### P0-2. 示例报告

至少准备 1 个完整示例，最好 2 个不同品类。

示例报告必须包含：

1. 一句话结论；
2. 自有 vs 对标对比表；
3. Top 3 问题；
4. 具体改法；
5. 收入影响情景；
6. 证据和限制说明。

示例可使用虚构品牌或已获授权的公开页面，但不能让用户误以为这些数字是真实后台数据。

### P0-3. 自有链接输入

支持填写：

- 自有首页 URL；
- 自有核心商品页 URL；
- 可选：广告落地页 URL。

校验：

- URL 格式正确；
- 页面可访问；
- 页面不是后台、登录页或私人链接；
- 失败时给出清晰原因。

### P0-4. 对标链接输入

支持 1–3 个对标页面，至少要求 1 个。

用户填写：

- 对标页面 URL；
- 选择对标理由：价格相近、商品相似、受众相似、品牌定位相似或其他。

Demo 不负责判断“真正竞争对手”，但要提醒用户：对标对象最好在价格、市场和受众上相近。

### P0-5. 基础经营数据输入

为了让收入情景不脱离实际，至少收集：

- 月访问量；
- 最近 30 天订单数；
- 平均客单价；
- 目标市场和币种。

建议加收：

- 毛利率；
- 主要流量来源；
- 移动端访问占比；
- 加购数和 checkout 数。

支持“我暂时不知道”，但此时必须禁止输出伪精确收入数字，只输出需要补充的数据清单。

### P0-6. 公开页面转化要素抽取

第一版围绕以下 10 类能力：

1. 首屏价值主张和目标人群；
2. 产品卖点和解决的问题；
3. 使用场景和适用对象；
4. 价格、折扣和套装；
5. CTA 与购买路径；
6. 评价、UGC 和社会证明；
7. 配送、运费、退货和质保；
8. FAQ、尺寸/材质/成分等购买疑虑；
9. 支付、联系方式、认证和品牌信任；
10. 订阅、复购、交叉销售和售后触达。

每一项输出三种状态：

- 已发现；
- 未发现；
- 未验证。

“未发现”不能直接等于“页面没有”，必须保留证据和抓取范围。

### P0-7. 对比矩阵

核心结果必须能回答：

| 分类 | 含义 |
|---|---|
| 你有、对标也有 | 基础能力，不自动判定为问题 |
| 你没有、对标有 | 候选缺口 |
| 你有、对标没有 | 可能是你的优势 |
| 你有但表达弱 | 信息存在但不够前置/清晰 |
| 双方都没有 | 机会假设，需要测试 |
| 未验证 | 页面无法确认，不给结论 |

每个候选问题都必须能点回页面证据。

### P0-8. Top 3 问题和具体改法

Demo 不能只输出“建议增强信任”“建议优化文案”。每条必须回答：

- 用户可能担心什么；
- 当前页面哪里没有解决；
- 对标页面用了什么方式；
- 具体改哪一个页面位置；
- 文案、模块或素材怎么改；
- 改完观察哪个指标；
- 工作量是低、中还是高。

示例：

> 商品页首屏没有说明预计到货时间，用户需要进入配送政策页才能确认。建议在价格和 CTA 下方增加“预计 X–Y 个工作日送达”和退货入口。先观察商品页到加购率和 checkout 发起率。

### P0-9. 收入影响情景

用户提供数据后显示：

```text
当前收入 = 月访问量 × 当前转化率 × AOV
增量收入 = 月访问量 × 假设性转化率提升 × AOV
```

三档情景：

- 保守：当前 CVR × 1.10；
- 中性：当前 CVR × 1.20；
- 乐观：当前 CVR × 1.35。

必须同时显示：

- 输入数据；
- 计算公式；
- 情景假设；
- 是否扣除毛利、广告、物流和退货；
- “不是预测或保证”的说明。

没有访问量、订单数和 AOV 的一致口径时，只显示“暂不能估算”。

### P0-10. 线索收集和反馈

提交后收集：

- 邮箱或联系方式；
- 是否愿意参加 15–20 分钟访谈；
- 是否愿意提供 Shopify/GA4 截图；
- 是否允许匿名作为案例；
- 最想解决的问题。

不要强制用户立即注册账号。早期目标是收集有效商家，而不是堆注册数。

### P0-11. 人工复核后台

哪怕只是一个内部表格，也需要记录：

- 线索状态；
- 页面抓取状态；
- 对标选择是否合理；
- 抽取结果；
- 证据链接和截图；
- 报告状态；
- 用户反馈；
- 是否愿意付费。

### P0-12. 合规和失败提示

必须处理：

- 页面不可访问；
- 动态内容未加载；
- 页面需要登录；
- 触发验证码或反爬；
- 用户提交后台链接；
- 公开页面信息不足。

明确说明：

- 只分析公开页面；
- 不获取竞品后台数据；
- 不绕过验证码、登录或访问限制；
- 不保证转化率或收入增长；
- 页面差异是转化假设，需要自有数据和测试验证。

## 5. P1：有余力再做的能力

这些能力会提升体验，但不应阻塞第一版：

1. 用户上传 Shopify/GA4/Meta Ads CSV；
2. 自动识别移动端/桌面端页面截图；
3. 自动生成首屏文案改写版本；
4. 自动生成 FAQ、配送说明和信任模块草稿；
5. 支持多个 SKU 批量对比；
6. 报告链接和版本管理；
7. 7–14 天后的复测提醒；
8. 简单的用户反馈评分；
9. 多语言报告（中文/英文）；
10. 按品类展示常见转化要素模板。

---

## 6. P2：暂时不做

- Shopify OAuth；
- 实时订单/广告/利润同步；
- 自动修改 Shopify 主题；
- 自动部署 A/B 测试；
- 自动投放和优化广告；
- 竞品真实转化率、订单量和广告 ROI 推断；
- 账号体系、团队协作和订阅计费；
- 大规模行业基准数据库；
- 全渠道归因；
- 医疗、法律、税务、认证结论。

## 7. Demo 的最小页面结构

### 页面 1：Landing Page

区块：

1. 核心价值主张；
2. 适用商家；
3. 输入与输出说明；
4. 示例报告截图；
5. 证据和合规边界；
6. CTA：申请一次测试；
7. FAQ。

### 页面 2：提交表单

分成 3 步：

1. 店铺和对标链接；
2. 基础经营数据；
3. 联系方式与研究授权。

表单完成后不要马上声称“分析完成”，显示：

> 已收到，我们会先检查页面可访问性和对标匹配度，再生成初步结果。

### 页面 3：分析状态

显示 4 个阶段：

```text
检查链接
→ 提取公开信息
→ 对比转化要素
→ 人工复核报告
```

首版可以由人工更新状态，不必真实搭建队列系统。

### 页面 4：结果预览

只展示：

- 一句话结论；
- Top 3 问题；
- 一个对比矩阵；
- 一个收入情景卡片；
- “领取完整报告/预约解读”按钮。

### 页面 5：反馈/预约

询问：

- 哪一条建议最有价值；
- 哪一条不准确；
- 是否愿意修改；
- 是否愿意支付完整报告；
- 是否愿意参加访谈。

## 8. 推荐技术方案：先简单，别过度工程化

### 最小可用组合

- 前端：静态 HTML 或轻量 React；
- 表单：Tally 或自建表单；
- 数据：SQLite/CSV；
- 页面采集：合规公开网页读取；
- 抽取：Python + 结构化 JSON；
- 计算：Python/pandas；
- 报告：Markdown → HTML/PDF；
- AI：生成结构化抽取初稿和建议初稿；
- 人工后台：表格或简单管理页面。

### 关键数据结构

```json
{
  "lead": {
    "own_urls": [],
    "benchmark_urls": [],
    "category": "",
    "market": "",
    "sessions_30d": null,
    "orders_30d": null,
    "aov": null,
    "gross_margin": null
  },
  "evidence": [
    {
      "dimension": "shipping",
      "site": "own",
      "status": "weak_or_missing",
      "quote": "",
      "url": "",
      "captured_at": "",
      "confidence": "medium"
    }
  ],
  "recommendations": [],
  "revenue_scenarios": {}
}
```

## 9. Demo 验证指标

### 用户获取

- 50 条个性化触达；
- ≥10 个有效回复；
- ≥5 个商家提交真实链接；
- ≥3 个商家愿意提供更多数据或访谈。

### 产品价值

- ≥80% 的测试用户能复述 Top 3 问题；
- ≥60% 认为对比结果比单站建议更有帮助；
- ≥50% 表示至少有一条建议会执行；
- ≥1–2 个用户愿意支付 ¥99–¥299。

### 交付效率

- 一份轻量报告在 2 小时内完成；
- 页面分析失败时，100% 明确提示未验证；
- 收入公式和数据口径错误率为 0；
- 每条结论都有证据或明确标为假设。

### Go / No-Go

继续产品化的信号：

- 5 个有效商家中至少 3 个认可问题；
- 至少 2 个完成页面改动；
- 至少 1 个完成复测；
- 至少 1 个愿意付费；
- 重复出现的诊断维度可以模板化。

不继续开发的信号：

- 只有小白用户愿意参与；
- 成熟商家不愿提供任何数据；
- 用户只想找代运营，不需要诊断；
- 用户觉得建议正确但不值得花钱；
- 每单都需要完全不同的分析，无法标准化。

## 10. 推荐实施顺序

### 第 1 阶段：演示和线索收集

做：

- Landing Page；
- 1 个完整示例报告；
- 申请表；
- 内部线索表；
- 人工交付流程。

不做：

- 真实自动抓取全链路；
- 用户账号；
- 付费系统；
- 实时数据同步。

### 第 2 阶段：真实链接分析

做：

- 用户提交链接；
- 你手动触发分析；
- 生成真实对比结果；
- 收集反馈。

### 第 3 阶段：半自动化

做：

- 自动抽取稳定字段；
- 自动生成对比矩阵；
- 自动计算收入情景；
- 人工只复核证据和排序。

### 第 4 阶段：决定是否做 SaaS

只有出现重复需求、付费意愿和复测数据后，才做账号、数据同步和订阅。

---

## 最终建议

Demo 第一版只需要让用户完成一件事：

> **提交自己的页面和对标页面，并获得一份可信、具体、能执行的差异诊断。**

如果这个价值成立，后续可以不断自动化；如果这个价值不成立，做再漂亮的 SaaS 也只是把错误方向包装得更好看。
