# 竞品对标转化诊断 Demo PRD

> 版本：v0.2
> 目标：用于第一轮真实商家招募、用户访谈和付费意愿验证。
> 形态：前端可操作 Demo + 英文优先的 Mock 报告 + 公开页面分析 + 人工复核，不是完整 SaaS。
> 语言策略：英文为默认语言，中文作为切换选项；所有对外分享素材优先使用英文。

## 1. 产品结论

Demo 的首要任务不是自动化，而是验证：

- 有流量的独立站商家是否愿意提交自己的商品页和对标页面；
- “自有 vs 对标”的对比是否比普通单站建议更有价值；
- 用户是否愿意根据建议改页面并接受复测；
- 用户是否愿意为完整诊断支付 ¥99–¥299。

用户核心价值：

> 不只是告诉商家页面哪里不好，而是指出用户关心、对标页面已经处理、但商家自己没有做好或表达不清的地方，并给出下一步怎么改。

## 2. Problem Statement

Shopify 和独立站商家经常知道访问量、广告点击和订单，却不知道用户为什么没有继续购买。单看自己的页面，商家缺少判断标准；单看竞品，又容易停留在“对方设计更好看”的主观评价。

现有工具主要解决数据记录、行为回放或归因看板，不直接回答“你的页面和对标页面相比缺了什么、先改哪三件事、怎么验证”。Demo 用公开页面对比和人工复核，把差异转成可执行的转化假设。

### 目标用户

- Shopify/独立站卖家或 1–5 人 DTC 团队；
- 近 30 天有至少 1,000 次访问，优先 3,000+；
- 已投放 Meta、Google、TikTok，或有稳定自然流量；
- 有 1–3 个主推商品；
- 正在考虑优化商品页、首页或广告落地页。

### 不适合的用户

- 尚未开店或没有流量的新手；
- 只想找爆品/供应商的人；
- 需要企业级归因或自动投放的成熟品牌；
- 要求获取竞品后台、订单或广告账户数据的人。

## 3. Goals

### 用户目标

1. 用户能在 5–10 分钟提交自己的商品页、1–3 个对标页和基础经营数据；
2. 用户能看到一份结构清晰的对比结果，而不是泛泛建议；
3. 每个 Top 问题都有页面证据、具体改法和复测指标；
4. 用户能理解收入情景的计算方式和限制；
5. 用户能留下反馈，并选择是否参加访谈或复测。

### 验证目标

- 50 条个性化触达获得至少 10 个有效回复；
- 至少 5 个商家提交真实链接；
- 至少 3 个商家愿意提供额外数据或参加访谈；
- 至少 1–2 个用户愿意支付 ¥99–¥299；
- 至少 1 个用户完成改动并进入复测。

## 4. Non-goals

1. 不做完整 SaaS 账号体系、权限和订阅计费；
2. 不做 Shopify OAuth、实时订单同步和广告归因；
3. 不自动修改店铺，不自动部署 A/B 测试；
4. 不推断竞品真实转化率、订单量、广告 ROI 或利润；
5. 不绕过登录、验证码、反爬和访问限制；
6. 不保证转化率或收入增长；
7. 不提供法律、税务、医疗或认证结论。

## 5. 用户故事

### P0

- 作为有流量的独立站卖家，我希望输入自己的商品页和对标页面，以便找到页面之间的关键转化差异。
- 作为卖家，我希望看到价格、CTA、评价、配送、退货、FAQ、规格、信任和复购等要素的对比，以便知道用户可能在意什么。
- 作为卖家，我希望每条问题都落到具体页面位置和改法，以便本周能执行。
- 作为卖家，我希望看到访问量、订单数和 AOV 对收入情景的影响，以便判断改动优先级。
- 作为卖家，我希望知道哪些结论是页面事实、哪些只是需要测试的假设，以便避免误判。

### P1

- 作为卖家，我希望上传 Shopify/GA4/广告截图或 CSV，以便用后台漏斗验证页面假设。
- 作为卖家，我希望在 7–14 天后重新提交数据，以便观察改动前后变化。
- 作为卖家，我希望获得文案、FAQ 或信任模块草稿，以便减少执行成本。

### 异常场景

- 作为用户，我希望链接不可访问时看到明确原因，而不是收到编造的分析。
- 作为用户，我希望未提供经营数据时看到“暂不能估算”，而不是一个伪精确的增量收入。
- 作为用户，我希望知道平台只分析公开页面，不会索取 Shopify 密码或竞品后台权限。

## 6. 核心用户流程

```text
访问落地页
→ 查看示例报告
→ 提交自有链接和对标链接
→ 填写基础经营数据
→ 系统检查链接
→ 公开信息抽取
→ 对比转化要素
→ 人工复核
→ 查看结果预览
→ 留下联系方式领取完整报告
→ 提交反馈/预约访谈/进入复测
```

第一版允许手动触发分析和人工更新状态。前端展示统一流程即可，不要求真正搭建任务队列。

## 7. 页面和能力说明

### 7.0 分享版 Mock 报告（新增 P0）

Demo 必须提供一份不依赖真实用户数据的英文 Mock 报告，供 Reddit、Shopify Community、DTC 社群和私信分享。Mock 数据必须明确标注为 fictional/sample data，避免被误解为真实客户案例。

#### Mock 报告目标

- 让潜在商家在 30–60 秒内看到最终交付长什么样；
- 把抽象的“竞品对标”变成可理解的 insight card；
- 展示“发现 → 改法 → 指标 → 收入情景”的完整链条；
- 让用户可以复制链接、截图或转发给团队成员。

#### Mock 报告建议内容

- 虚构品牌：Northstar Pet Co.；
- 品类：premium dog walking gear；
- 自有页面：Northstar Harness；
- 对标页面：Trail & Tail Everyday Harness；
- 月访问量：3,000 sessions；
- 近 30 天订单：10；
- AOV：$120；
- 当前收入：$1,200；
- 中性情景：相对 CVR 提升 20%，情景增量收入 $240；
- 一句话结论：Your biggest leak is purchase confidence, not traffic；
- Top 3：配送承诺前置、场景化评价、套装价值表达；
- 每条包含页面证据、具体动作、复测指标和置信度。

#### 对外分享规则

- 标题和正文默认英文；
- 明确写 Sample report / fictional data；
- 不使用真实品牌或未经授权的截图作为“客户案例”；
- 不暗示对标品牌真实转化率更高；
- 收入数字只用于演示模型，不是预测或保证；
- 任何真实客户结果必须获得匿名和展示授权。

### 7.6 双语能力与国际受众（新增 P0）

- 默认语言为英文；
- 提供 EN / 中文切换；
- 语言切换不改变表单数据和当前页面状态；
- 英文字段、CTA、免责声明、示例报告文案完整，不使用机器直译痕迹；
- 英文金额默认使用 USD，示例报告使用美国商家熟悉的 sessions、orders、AOV、CVR、add-to-cart、checkout 等术语；
- 中文版本用于中国跨境卖家，金额和术语可按中文表达展示；
- 分享卡片默认英文，并保留复制按钮。


### 7.1 概览页 / Landing Page

#### 目的
在 30 秒内让目标商家理解 Demo 的价值、输入和输出。

#### 页面内容

- 标题：找出用户为什么更愿意买对方；
- 副标题：输入自己的商品页和 1–3 个对标页面，获得转化要素差异、最优先改动和收入影响情景；
- 输入/输出卡片；
- 示例结论；
- 申请一次对比诊断按钮；
- 查看示例报告按钮；
- 公开页面、人工复核、不保证结果等说明。

#### 验收标准

- [ ] 首屏能说明目标用户、输入、输出和不需要 Shopify 登录；
- [ ] 用户可从概览进入提交页或示例报告；
- [ ] 页面明确标注早期测试版和公开页面边界。

### 7.2 提交诊断页

#### 输入字段

| 字段 | 必填 | 说明 |
|---|---|---|
| 自有商品页 | 是 | 核心 SKU 的公开商品页 |
| 自有首页 | 否 | 用于品牌和首屏补充分析 |
| 对标页面 1 | 是 | 至少 1 个 |
| 对标页面 2 | 否 | 最多 3 个对标页 |
| 月访问量 | 建议 | 需标明 sessions/users 口径 |
| 近 30 天订单数 | 建议 | 与访问量同一时间窗口 |
| 平均客单价 | 建议 | 用于收入情景 |
| 目标市场/币种 | 是 | 影响文案、配送和金额展示 |
| 当前困扰问题 | 是 | 用于理解上下文 |
| 邮箱 | 是 | 接收报告和研究邀请 |

#### 验收标准

- [ ] 自有商品页和至少一个对标页为空时不能提交完整诊断；
- [ ] 支持用户暂时不知道访问量、订单数或 AOV；
- [ ] 缺少数据时不生成伪精确收入数字；
- [ ] 提交后显示“已收到，下一步检查页面可访问性”；
- [ ] 不索取 Shopify 密码、支付凭证或竞品登录信息。

### 7.3 分析状态页

#### 状态

```text
检查链接
→ 提取公开信息
→ 对比转化要素
→ 人工复核报告
```

第一版可由内部人工更新状态，用户只看到统一的进度文案。

#### 异常状态

- 页面不可访问；
- 页面需要登录；
- 动态内容未加载；
- 触发验证码/反爬；
- 对标页面与品类明显不匹配；
- 页面信息不足。

异常时必须显示：

1. 发生了什么；
2. 哪些结论不能生成；
3. 用户可以提供什么替代信息，例如公开截图；
4. 不会绕过访问限制。

### 7.4 结果预览页

#### 结果结构

1. 一句话结论；
2. 证据等级和抓取时间；
3. 当前月收入和中性情景增量（若数据完整）；
4. Top 3 改动；
5. 对比摘要；
6. 收入估算限制；
7. 反馈与复测 CTA。

#### 转化要素维度

- 首屏价值主张和目标人群；
- 产品卖点、问题和使用场景；
- 价格、折扣、套装、订阅；
- CTA 和购买路径；
- 评价、UGC 和社会证明；
- 配送、运费、退货、质保；
- FAQ、尺寸、材质、成分和规格；
- 支付方式、认证、联系方式和品牌信任；
- 复购、交叉销售和售后触达。

#### 对比分类

| 分类 | 解释 |
|---|---|
| 你有、对标有 | 双方均具备，不自动判定问题 |
| 你没有、对标有 | 候选缺口，需要结合场景排序 |
| 你有、对标没有 | 可能是自有优势 |
| 你有但表达弱 | 信息存在，但不够前置或清楚 |
| 双方都没有 | 机会假设，需要用户研究或测试 |
| 未验证 | 页面无法确认，不输出确定结论 |

#### Top 3 建议格式

每条必须包含：

- 问题陈述；
- 用户可能顾虑；
- 页面证据；
- 对标差异；
- 页面位置；
- 具体改法；
- 工作量；
- 复测指标；
- 置信度。

### 7.5 反馈与复测页

字段：

- 最有价值的部分；
- 哪一条建议会执行；
- 是否愿意提供后台截图/CSV；
- 是否愿意参加 15 分钟访谈；
- 是否愿意购买完整报告；
- 其他建议。

## 8. 收入情景模型

### 公式

```text
当前转化率 = 近 30 天订单数 ÷ 月访问量
当前收入 = 月访问量 × 当前转化率 × AOV
增量收入 = 月访问量 × 预计转化率提升 × AOV
```

### 三档情景

- 保守：当前 CVR 相对提升 10%；
- 中性：当前 CVR 相对提升 20%；
- 乐观：当前 CVR 相对提升 35%。

### 约束

- 只在访问量、订单数、AOV 口径一致时计算；
- 结果为数学情景，不是预测或保证；
- 不默认扣除广告、物流、退款和客服成本；
- 如果用户提供毛利率，可额外展示贡献利润情景；
- 公开页面无法证明真实转化因果，必须显示限制说明。

## 9. 后台人工复核

第一版用表格或 SQLite 即可，记录：

- lead_id；
- 提交时间；
- 自有 URL 和对标 URL；
- 品类、市场和数据口径；
- 链接状态；
- 证据摘要、来源 URL、抓取时间；
- 要素抽取结果；
- Top 3 建议；
- 收入模型输入和输出；
- 报告状态；
- 用户反馈、付费意向和复测状态。

AI 只生成抽取和建议初稿。人工必须复核链接、证据、计算、优先级和合规边界后再交付。

## 12. 指标和判断门槛

### 用户行为

- 目标商家落地页到表单开始率 ≥ 20%；
- 表单完成率 ≥ 50%；
- 提交链接后有效分析率 ≥ 80%；
- 至少 5 个真实商家提交链接。

### 价值验证

- 80% 测试用户能复述 Top 3 问题；
- 60% 认为对比结果比单站建议更有帮助；
- 50% 表示至少会执行一条建议；
- 1–2 个用户愿意支付 ¥99–¥299。

### 交付质量

- 轻量报告 2 小时内完成；
- 结论证据覆盖率 100%；
- 未验证项标注率 100%；
- 计算错误率 0%；
- 页面抓取失败时不编造结论。

### Go / No-Go

继续产品化：

- 5 个有效商家中至少 3 个认可问题；
- 至少 2 个用户完成页面改动；
- 至少 1 个用户完成复测；
- 至少 1 个用户付费；
- 至少一个诊断维度可重复模板化。

暂缓开发：

- 只有小白用户响应；
- 成熟商家不愿提供任何数据；
- 用户认为建议有道理但不值得付费；
- 用户真正需要的是代运营而非诊断；
- 每单都无法标准化。

## 13. 交付与时间线

### 第 1 阶段：原型

- 完成落地页、提交表单、示例报告、反馈页；
- 用虚构数据演示交互；
- 开始收集目标商家线索。

### 第 2 阶段：真实人工诊断

- 选择 3–5 个真实店铺；
- 手动分析链接；
- 交付报告；
- 访谈并记录用户反应。

### 第 3 阶段：半自动化

- 自动抽取稳定字段；
- 自动生成对比矩阵；
- 自动计算收入情景；
- 人工只复核证据、优先级和异常。

### 第 4 阶段：是否开发 SaaS

只有在有重复需求、真实付费和复测数据后，才进入账号、数据同步和订阅阶段。
